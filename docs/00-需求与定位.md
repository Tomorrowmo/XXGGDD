> 定位层面的结论已与决策方确认锁定；实现层面的细节在后续设计文档中展开。

## 0. 一句话定位

> **一个面向甲方评估方的「组合动力仿真/实验数据智能评估平台」——把多家承研单位（北航、西工大等）提交的仿真与实验数据统一入库、横向对比、智能量化分析，并借助 AI 产出可复用的评估结论与报告。**

落点四词：**评估**（非展示）· **多源对比**（非单家）· **智能量化**（非人工看图）· **可复用**（非一次性硬编码）。

**核心价值主张 = 评估/评判**（已确认锁定）。平台的魂是帮甲方回答"哪家更可信、更优、依据是什么"。分析工具（云图、曲线、壁面力…）是**评估的证据来源**，不是终点。

### 定位差异（本次重构的根本动机）

| | 现原型（西工大） | 目标平台 |
|---|---|---|
| 本质 | 单家数据的"展示 + 解析 + 总结"工具 | 多家数据的"评估 + 对比 + 智能分析"系统 |
| 核心动作 | 上传 → 看图 | 选择评估对象 → 看对比 → 得结论/报告 |
| 数据 | 临时上传、无持久化 | 批量入库、可比、可下钻 |

## 1. 用户与场景

| 维度 | 内容 |
|------|------|
| **核心用户** | 甲方评估人员（需客观评判各单位交付物优劣的人） |
| **上游输入** | 各单位提交的 ①仿真数据（Fluent/Front HDF5 等）②实验数据（热试车 TXT） |
| **核心诉求** | "哪家的数据更可信、结果更优？依据是什么？"——需**量化、可对比、可解释** |
| **交付产出** | 对比分析视图 + 智能诊断结论 + 可导出的评估报告 |
| **节奏约束** | 以**周**为单位输出可见进展，满足阶段性汇报 |

## 2. 现状能力盘点 vs 核心差距

### 现原型已具备（重构要保留的地基）
- 数据解析：实验 TXT、Fluent HDF5 后处理
- 三维/二维可视化：对称面云图、沿程面平均曲线等
- 结果提取：壁面力积分、入口参数、通道统计
- 初步 AI：LLM 诊断、VLM 云图分析（**但多为写死/单场景**）

> 注：现有能力"不一定是功能，好多是写死的"。真实能力边界须以代码家底摸查为准（见 `docs/01-现状代码能力盘点.md`）。

### 七个核心差距（= 重构要补的能力）

| # | 现状痛点 | 目标能力 |
|---|---------|---------|
| 1 | 单家数据展示 | **多源并行对比**（勾选多家/多算例交叉比对） |
| 2 | 临时上传、无持久化 | **批量入库 + 数据库存储** |
| 3 | 前端直接渲染大文件、性能差 | **后端预处理/沙箱**，可视化提效 |
| 4 | 无自动化报告 | **AI 驱动的报告与图表生成** |
| 5 | 无登录/会话/多用户 | **用户会话与协作管理** |
| 6 | 分析流程不可复用 | **可复用、可勾选历史、可配置的分析流程** |
| 7 | 算法逻辑写死在前端 | **算法模块化 → 可调用能力（skills）** |

## 3. 四大目标方向（会议主线）

1. **多数据批量入库入口** —— 替代手动上传，支撑对比的前提
2. **可视化效率提升** —— 后端预处理/沙箱，缓解前端压力
3. **实验数据智能量化分析** —— 从"看图"到"AI 自动诊断 + 量化打分"
4. **AI 可复用分析能力** —— AI coding 动态生成脚本 → 逐步沉淀为 skills

---

## 4. 分层对比模型（架构关键约束）

对比是**分层的**（已确认）：既要按单位/批次整体评级，也要按工况点做硬核量化。数据库不能只有单一主键，须是层级结构。

```
评估对象层级（对比可发生在任一层）
│
├─ L1 单位 / 交付批次   ← "北航这次交付 vs 西工大这次交付，整体质量谁高"（结论层，甲方汇报用）
│   └─ L2 算例 / 试验车次   ← "同一单位不同算例的一致性"（聚合与下钻）
│       └─ L3 工况点        ← "同一工况下，各家仿真值 vs 实验真值的偏差"（最细、最硬的量化证据）
│
└─ 横切维度：物理量（压力/温度/壁面力…）· 时间/收敛状态 · 数据来源类型（仿真/实验）
```

- **L3 工况点对齐**是量化评估的**硬核**——同一工况下各家偏差，才是"谁更准"的直接证据
- **L1 单位/批次**是甲方要的**结论层**——整体优劣评级
- 中间层做聚合与下钻

> **架构含义**：数据入库时必须打上 `单位 / 交付批次 / 算例 / 工况标识 / 物理量 / 数据类型` 标签，才能支撑任意层级对比。这套**评估元数据模型**是阶段一必须先定、最忌返工的地基（见 `docs/02-评估元数据模型.md`）。

## 5. 分阶段目标（先能跑，再变强，最后固化）

**阶段一 · 基础能力建设**
批量导入入库替代手动上传 · 复用 C-pass 服务做解析（提稳定性）· 借 Wu Cheng 方案做可视化性能优化（沙箱/后端预处理）。
→ 关键交付：**评估元数据模型 + 批量入库 + 可视化提效**。

**阶段二 · 智能化分析增强**
AI 对实验数据初步智能诊断 + 图表生成 · 接入 AI coding 动态生成分析脚本（**先不急着建完整 skills 体系**）· 复用现有仿真 HDF5 解析逻辑呈现结果。

**阶段三 · 算法迁移与流程固化**
提取西工大写死的算法（边界参数提取、壁面力计算等）· 支持**自定义分析流程配置，去硬编码** · 沉淀为可迁移复用的标准流程/技能组件。

## 6. 贯穿始终的设计红线

- **UI 延续性**：保留标题"组合动力智能评估"、RAGFlow 对话区、"仿真分析/实验分析"模块位置、左侧导航结构；右侧内容区**逐步替换增强**，不推倒重来
- **先设计后开发**：架构先具备扩展性，再动手
- **并行推进**：一边做系统设计，一边由胡飞团队提取现有算法
- **可持续迭代**：架构要能承接后续单位、后续数据类型接入
- **技术框架**：整体框架考虑采用 **DeepAgents 或借鉴其思想**（Agent 驱动、可复用 skills、动态编排）
