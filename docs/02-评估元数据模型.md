# 评估元数据模型（L1–L3 层级 + 横切维度）

> 平台的地基。一切"多源对比、量化打分、下钻追溯"都由这套模型支撑。
> 设计原则：**可比性优先**——任何入库数据都必须能被拉到同一标尺下比较。
> 借鉴 SimGraph 已验证的"四级置信度"机制与 sim-knowledge 的判据/映射结构。
> 状态：设计草案 v0.1，含若干待决假设（见文末 §7）。

---

## 0. 设计要回答的核心问题

1. 同一发动机工况下，北航仿真值 / 西工大仿真值 / 实验实测值，怎么**对齐到一起**才能比？（← 最难，是模型的心脏）
2. 每个数据点的**可信度**怎么标注、怎么追溯？
3. "谁更优"的**打分**挂在哪一层、依据什么？
4. 甲方汇报要的**单位整体评级**怎么从底层量化聚合上来？

---

## 1. 实体总览（L1 → L2 → L3 + 参照系 + 打分）

```
┌─ L1 ────────────────────────────────────────────────────────┐
│  Unit 单位 ── has ──▶ Delivery 交付批次                        │
│  (北航/西工大)         (某单位某次提交)                          │
└──────────────────────────────────┬───────────────────────────┘
                                    │ contains
┌─ L2 ───────────────────────────── ▼ ─────────────────────────┐
│  Case 算例/车次  { kind: simulation | experiment }            │
│  (一个 Fluent 算例 / 一次热试车)                                │
└──────────────────────────────────┬───────────────────────────┘
                                    │ produces + aligned_to
┌─ L3 ───────────────────────────── ▼ ─────────────────────────┐
│  Measurement 测量  ──▶  OperatingPoint 工况点 (canonical) ◀── │
│  (某物理量的一个值)      (共享参照身份，跨单位/跨来源对齐的锚)      │
└──────────────────────────────────────────────────────────────┘
        ▲                                   ▲
        │ 横切                               │ 参照系
   Quantity 物理量登记         Reference/GroundTruth 真值(实验或标准算例)
   Confidence 置信度           Provenance 证据链
        │
   Evaluation 评估打分 (挂在 L3 measurement / L2 case / L1 delivery 三层聚合)
```

## 2. 核心实体定义

### L1 层

**Unit（单位）**
| 字段 | 说明 |
|---|---|
| `unit_id` | 主键 |
| `name` | 北航 / 西工大 / … |
| `type` | 承研单位 / 甲方 / 第三方 |

**Delivery（交付批次）** —— 甲方评级的最小交付单元
| 字段 | 说明 |
|---|---|
| `delivery_id` | 主键 |
| `unit_id` | → Unit |
| `label` | 批次名（如"2026Q1 一轮交付"） |
| `submitted_at` | 提交时间（**由入库时写入，非脚本生成**） |
| `note` | 描述 |

### L2 层

**Case（算例 / 试验车次）** —— 仿真与实验的统一抽象
| 字段 | 说明 |
|---|---|
| `case_id` | 主键 |
| `delivery_id` | → Delivery |
| `kind` | `simulation` \| `experiment`（关键区分） |
| `name` | 算例名 / 车次号 |
| `source_format` | fluent-hdf5 / openfoam / cgns / txt-experiment / … |
| `storage_uri` | 文件存储位置 |
| `content_hash` | SHA-256，**入库去重**（借鉴 SimGraph） |
| `parse_status` | pending / parsed / failed |
| `parse_confidence` | HIGH / MED / LOW / PENDING（见 §4） |
| `context` | 求解器/湍流模型/燃料/网格量等（借鉴 sim-knowledge `context`） |

### L3 层 —— 心脏

**OperatingPoint（工况点，canonical 规范工况）** —— 跨单位/跨来源对齐的**共享锚**
| 字段 | 说明 |
|---|---|
| `op_id` | 主键 |
| `canonical_key` | 规范工况标识（平台统一定义，见 §3） |
| `params` | 定义该工况的物理条件：如 `{Ma, 高度/来流压力, 总温, 燃料流量, 余气系数…}` |
| `description` | 人类可读描述 |

> 关键：OperatingPoint 是**平台侧的规范身份**，不属于任何单位。各单位的算例通过"映射"挂到它上面，这样"同一工况"才有唯一定义。

**Measurement（测量）** —— 一个物理量的一个值
| 字段 | 说明 |
|---|---|
| `meas_id` | 主键 |
| `case_id` | → Case（来自哪个算例/车次） |
| `op_id` | → OperatingPoint（对齐到哪个规范工况） |
| `quantity_id` | → Quantity（是什么物理量，见横切） |
| `value` / `unit` | 值 + 量纲（**经语义归一后的标准量纲**） |
| `raw_name` | 原始列名/变量名（保留可追溯） |
| `source_kind` | simulation \| experiment（冗余 Case.kind，便于对比查询） |
| `status` | 判据触发状态（normal / warning / anomaly，来自判据引擎） |
| `confidence` | HIGH / MED / LOW / PENDING |
| `evidence` | 证据链（见 Provenance） |

## 3. 心脏：工况对齐机制（最难、最忌返工）

问题：各单位数据里的工况标识**不一定一致**（命名不同、字段不同、甚至只能从数据推断）。直接用某单位的字段做主键 → 对不齐。

**方案：Canonical OperatingPoint 注册表 + 映射层（三级来源标注）**

```
Case (某单位某算例)
   │  aligned_to  { method, mapping_confidence }
   ▼
OperatingPoint (平台规范工况)
```

**CaseOperatingLink（算例↔工况映射）**
| 字段 | 说明 |
|---|---|
| `case_id` / `op_id` | 关联 |
| `method` | `auto`（从数据自动识别）\| `rule`（规则/映射表）\| `manual`（人工指定） |
| `mapping_confidence` | HIGH / MED / LOW / PENDING |
| `mapped_by` | 系统 / 规则名 / 用户 |

- 入库时先尝试 `auto`/`rule` 对齐；对不上的落 `PENDING`，进**人工待办队列**由评估人员确认（借鉴 SimGraph 的 PENDING 兜底 + cfd-knowledge-review 的人工闸门）。
- 好处：无论各单位工况怎么标，最终都收敛到平台统一的 `canonical_key`，且**对齐动作本身带置信度、可追溯、可人工纠偏**。

> ⚠️ `canonical_key` 到底由哪些物理量定义、各单位数据里有没有可用来自动对齐的字段——是本模型**唯一真正阻塞、需与业务确认**的点（见 §7-1）。模型结构对此已解耦：无论答案如何，映射层都兜得住。

## 4. 横切维度

### 4.1 Quantity（物理量登记）—— 直接复用 sim-knowledge `mappings`
统一物理量语义，解决"各单位变量名不同 / 同名不同义 / 量纲不一"。
| 字段 | 说明 |
|---|---|
| `quantity_id` | 主键（如 `wall_pressure`、`thrust`、`mass_flow`） |
| `physical_name` / `standard_unit` | 规范名 + 标准量纲 |
| `aliases` | raw_name 映射表（OpenFOAM `p`=运动压力 m²/s²≠Pa 这类坑，直接用 `mappings/ambiguity_table`） |
| `valid_range` | 值域（用于合理性判据） |

### 4.2 Confidence（四级置信度）—— 直接复用 SimGraph 机制
`HIGH`（判据/真值直接命中）· `MED`（规则推断）· `LOW`（弱证据）· `PENDING`（待人工）。
贯穿三处：`Case.parse_confidence`（解析质量）、`CaseOperatingLink.mapping_confidence`（对齐质量）、`Measurement.confidence`（数值可信度）。

### 4.3 Provenance（证据链）
每个 Measurement / Evaluation 可追溯到：源文件+位置、解析工具+版本（如 simparse tier）、触发的判据 intent、引用的文献/真值。**评估结论可追溯是与"通用 CFD 助手"的核心差异化。**

### 4.4 Reference / GroundTruth（参照系）
| 来源 | 说明 |
|---|---|
| 实验真值 | 某工况的 `kind=experiment` Measurement 作为该工况的"真值"参照 |
| 标准算例真值 | sim-knowledge `labeled_cases`（NASA/ECN 等）5% 容差比对 |

## 5. 评估打分（三层聚合）

**Evaluation（评估记录）** —— 可挂在三个层级
| 层 | 评估内容 | 依据 |
|---|---|---|
| **L3 Measurement** | 某工况某物理量：各单位值 vs 真值的偏差、是否触发异常判据 | sim-knowledge `criteria` + `simparse_validate` |
| **L2 Case** | 算例整体：收敛性、网格质量、QOI 合理性、一致性 | 渊「专家判断」+ 判据聚合 |
| **L1 Delivery** | 单位批次整体评级（甲方汇报层） | L2/L3 加权聚合 |

| 字段 | 说明 |
|---|---|
| `eval_id` / `target_level` / `target_id` | 挂在哪层的哪个对象 |
| `score` / `verdict` | 量化分 / 结论（可信/存疑/异常） |
| `deviation` | 相对真值偏差（L3） |
| `criteria_hits` | 触发的判据 intent 列表（可追溯） |
| `confidence` / `evidence` | 置信度 + 证据链 |

> 评分**读同一份 sim-knowledge criteria**，不另写阈值。权重方案待定（§7-3）。

## 6. 这套模型直接支撑的对比查询（验证设计是否够用）

| 甲方问题 | 查询路径 |
|---|---|
| "同工况下各家压力谁更准？" | 固定 `op_id`+`quantity_id` → 拉所有 `source_kind=simulation` 的 Measurement，对比 `kind=experiment` 真值算 `deviation` |
| "北航这次交付整体 vs 西工大？" | 对比两 `delivery_id` 的 L1 Evaluation |
| "某单位不同算例一致吗？" | 同 `delivery_id` 下多 `case_id` 的 L2 Evaluation |
| "这个异常是哪来的？" | Measurement → Provenance → 源文件+判据+文献 下钻 |
| "自然语言问：找出所有壁面压力超范围的西工大算例" | NL → 结构化查询（借鉴 SimGraph NL2Cypher）|

**四类对比（L1/L2/L3 + 下钻）+ 自然语言检索，模型均可支撑** → 设计闭环。

## 7. 待决假设（不阻塞结构，但需业务确认）

1. **【阻塞对齐语义，最关键】** `canonical_key` 由哪些物理量定义？各单位交付数据里是否有可用于 `auto`/`rule` 自动对齐的一致字段，还是主要靠人工？—— 结构已用映射层解耦，但语义需业务给。
2. **单位/批次的唯一编号规则**由谁定、怎么编？
3. **L1 评级的评分维度与权重**：是既定标准，还是平台设计？
4. **存储技术选型**：图库（Neo4j，跟 SimGraph 一致，利于灵活元数据 + NL 检索）还是关系库（Postgres + JSON 元数据）？—— 归入 [04-目标架构与技术选型.md](04-目标架构与技术选型.md) 决策，不影响本模型的实体/关系设计。
5. **实验 vs 仿真的"真值"地位**：实验一定是真值，还是实验本身也要被评估可信度？（当前设计：实验既是被评对象，也可充当该工况参照——两用。）
